TOPIC 01

Process vs Thread, PCB & Context Switching

Core Concept
A process is a program in execution — an active entity with its own memory space, resources, PCB, and state. A thread is the smallest unit of CPU execution within a process. Multiple threads share the same process address space but execute independently.

Context Switching is the mechanism by which the OS saves the state of a running process/thread and loads the state of the next — it is pure overhead; no useful work is done during the switch itself.

ThreadLightweight unit of execution within a process; shares heap, code, data
PCBProcess Control Block — kernel data structure storing all process metadata

DispatcherModule that gives CPU control to the process selected by the scheduler
Dispatch LatencyTime taken to stop one process and start another
User ThreadManaged by user-level library (e.g., POSIX pthreads)

Real-World Analogy
Process = A Restaurant. The restaurant has its own kitchen (memory), staff (resources), and menu (code).

Thread = A Chef in the restaurant. Multiple chefs share the same kitchen (heap/data), but each works on their own dish (stack/PC).

Context Switch = Shift Change. When a chef leaves, they note exactly what stage the dish is at (save state to PCB), and the next chef reads those notes and continues (loads state from PCB).

Viva Q&A
Q: What is a process?
A process is a program in execution. It includes the program code, current activity (PC), stack, data section, and heap. Each process has its own PCB stored in kernel memory.
Q: What does the PCB contain?
PID, process state, program counter, CPU registers, memory management info (page tables), I/O status info, accounting info, scheduling info, PPID, and list of child PIDs.
Q: What is saved during a context switch?
The complete CPU state: program counter, ALL CPU registers (general-purpose + status flags), stack pointer, and memory management registers. Saved into the process's PCB by the OS kernel (dispatcher).
Q: Why are threads called lightweight processes?
Threads share the process's address space (code, heap, data, files). Only thread-specific data (PC, registers, stack) is separate. Creation, switching, and communication are all faster than full processes.
Q: Can two threads of the same process run truly in parallel?
Yes — on a multi-core CPU, two threads of the same process can run on separate cores simultaneously, achieving true parallelism.
Q: What is the difference between concurrency and parallelism?
Concurrency = multiple tasks making progress via time-sharing (single core possible). Parallelism = multiple tasks literally executing simultaneously (requires multi-core).
Q: Who does the saving in context switch — hardware or software?
Both. Hardware saves minimal state (PC, PSW) on interrupt. Software (OS kernel dispatcher) saves the full register state to the PCB.
Trap Questions ⚠
Q: Is a process with one thread the same as a thread?
No. A process always has a PCB, memory space, and OS resources. A thread has only its execution context. Even a single-threaded process carries process-level overhead that a raw thread doesn't.
Q: If threads share memory, why does each thread have its own stack?
Each thread has its own execution flow with its own function calls, local variables, and return addresses — all of which are stack-based. These must be independent; otherwise one thread's function call would corrupt another's.
Q: Is context switching done by the scheduler?
No! The scheduler SELECTS the next process (policy decision). The DISPATCHER performs the actual context switch (mechanism). Dispatch latency is the time the dispatcher takes to complete the switch.
Common Mistakes
"PCB is stored in process's own memory"
PCB is stored in KERNEL memory (process table)
"Context switch saves only the PC"
It saves the ENTIRE CPU state — all registers, flags, SP
"All threads have separate heaps"
Threads share the heap; only the STACK is separate
"Context switch is done by the scheduler"
Scheduler SELECTS; DISPATCHER does the actual switch
1-Minute Revision
  • Process = program in execution; has PCB, own memory, OS resources
  • Thread = unit of execution within process; shares code/heap/data
  • PCB stores: PID, state, PC, registers, memory info, I/O info (in KERNEL memory)
  • 5 Process States: New → Ready → Running → Waiting → Terminated
  • Context Switch: save CPU state to PCB → load next PCB → resume (pure overhead)
  • Done by DISPATCHER, not scheduler; scheduler only selects
  • Thread own: Stack, PC, Registers | Thread share: Code, Heap, Data, Files
  • User thread: fast switch, one blocks all | Kernel thread: true multicore
TOPIC 02

CPU Scheduling Algorithms

Core Concept
CPU scheduling decides which ready process gets the CPU next and for how long. The scheduler selects from the ready queue; the dispatcher performs the context switch. Goal: maximize CPU utilization and throughput while minimizing turnaround time, waiting time, and response time.
Key Terms
TermDefinition
CPU BurstPeriod of CPU execution between I/O requests
Arrival Time (AT)When process enters the ready queue
Burst Time (BT)Total CPU time the process needs
Completion Time (CT)When the process finishes execution
Turnaround Time (TAT)CT − AT (total time spent in system)
Waiting Time (WT)TAT − BT (time spent waiting in ready queue)
Response TimeTime from submission to FIRST CPU response
PreemptiveOS can forcibly take CPU from running process
Non-preemptiveProcess runs until it voluntarily gives up CPU
StarvationProcess never gets CPU due to continuous preemption by higher-priority processes
AgingGradually increasing priority of waiting processes to prevent starvation
Convoy EffectShort processes queue behind a long process in FCFS
TAT = CT − AT   |   WT = TAT − BT   |   Response Time = First_CPU_time − AT
Algorithm Breakdown
ALGORITHMS OVERVIEW
────────────────────────────────────────────────────────────────

1. FCFS (First Come First Served)
   • Non-preemptive; FIFO queue
   • Problem: CONVOY EFFECT — short processes wait behind long ones
   • No starvation

2. SJF (Shortest Job First) — Non-preemptive
   • Selects process with shortest BURST TIME
   • Optimal for minimizing average waiting time
   • Problem: Cannot know burst time in advance (estimated via exponential averaging)

3. SRTF (Shortest Remaining Time First) — Preemptive SJF
   • New process preempts current if its BT < current's remaining BT
   • Minimizes average waiting time among all algorithms

4. Round Robin (RR)
   • Each process gets a fixed TIME QUANTUM (q)
   • After quantum: process goes to back of ready queue
   • Best for time-sharing; prevents starvation
   • Large q → degenerates to FCFS | Small q → excessive context switches

5. Priority Scheduling
   • Lower number = higher priority (usually)
   • Preemptive or non-preemptive
   • Problem: STARVATION | Solution: AGING (increase priority over time)

6. MLFQ (Multilevel Feedback Queue)
   • Processes CAN move between queues
   • I/O-bound → promoted | CPU-bound → demoted
   • Used in real OS (Linux, Windows) — most flexible
Diagrams
MLFQ STRUCTURE
──────────────────────────────────────────────
Priority 1 (Highest) → [Q1: RR, q=8ms ] → CPU
         │                    │ (if not done in q)
         │                    ▼
Priority 2           → [Q2: RR, q=16ms] → CPU
         │                    │ (if not done in q)
         │                    ▼
Priority 3 (Lowest)  → [Q3: FCFS      ] → CPU

New process enters Q1. If it doesn't finish in q,
it drops to Q2, then Q3 → rewards I/O-bound, 
demotes CPU-hungry processes automatically.

Real-World Analogy
FCFS = Bank queue — whoever arrives first gets served first, regardless of transaction complexity.

SJF = Express checkout lane — person with fewest items goes first. Fastest overall, but unfair to large shoppers.

Round Robin = Rotating presidency — each country gets to speak for exactly 2 minutes at the UN, then the next. Fair, but overhead in switching.

Viva Q&A
Q: Which algorithm gives minimum average waiting time?
SJF (non-preemptive) and SRTF (preemptive) are provably optimal for minimizing average waiting time among all scheduling algorithms.
Q: What is the Convoy Effect?
In FCFS, one long CPU-bound process holds the CPU while short processes and I/O-bound processes queue behind it — like a slow truck blocking fast cars. Results in high average waiting time.
Q: What happens if time quantum in RR is very large/very small?
Very large → degenerates to FCFS (process completes before quantum). Very small → excessive context switches; CPU wastes all time switching and no useful work is done.
Q: How does MLFQ prevent starvation?
By implementing AGING — a process waiting too long at a lower-priority queue is moved up to a higher-priority queue over time, ensuring it eventually gets CPU time.
Q: Why can't SJF be implemented exactly in practice?
Because burst time is unknown in advance. OS estimates it using exponential averaging: τ(n+1) = α × t(n) + (1−α) × τ(n), where t(n) is actual burst and τ(n) is predicted burst.
Q: Difference between scheduler and dispatcher?
Scheduler selects which process runs next (policy). Dispatcher actually performs the context switch and hands CPU control (mechanism). Dispatch latency = time for dispatcher to complete the switch.
Trap Questions ⚠
Q: Does FCFS cause starvation?
NO. FCFS does NOT cause starvation — every process eventually gets the CPU. It causes the Convoy Effect, but that's poor performance, not starvation. Starvation is associated with Priority Scheduling and SJF.
Q: Is SJF always optimal?
Only optimal for average WAITING TIME, not necessarily turnaround time or response time. Also, it can starve long processes.
Q: In SRTF, can a newly arrived process preempt the running one?
Yes — if the newly arrived process has remaining burst time LESS than the current process's remaining time, it immediately preempts the current process.
Common Mistakes
"FCFS can cause starvation"
FCFS causes Convoy Effect; it does NOT cause starvation
TAT = CT (forgetting to subtract AT)
TAT = CT − AT always; WT = TAT − BT
"Larger quantum = better RR"
Larger quantum → better throughput but WORSE response time
"SJF gives optimal results always"
Optimal only for average WAITING TIME, not overall
1-Minute Revision
  • TAT = CT−AT | WT = TAT−BT | Response = First_CPU − AT
  • FCFS: simple, no starvation, Convoy Effect
  • SJF: optimal avg WT, non-preemptive, needs burst time knowledge
  • SRTF: preemptive SJF, best avg wait, high preemption overhead
  • Round Robin: fair, time quantum, prevents starvation
  • Priority: starvation risk → fix with AGING
  • MLFQ: best practical algorithm, used in real OS (Linux/Windows)
  • Convoy Effect = FCFS problem | Starvation = Priority/SJF problem
TOPIC 03

Process Synchronization & Critical Section

Core Concept
When multiple processes/threads access shared data concurrently, results can be incorrect due to race conditions. The Critical Section Problem requires designing protocols so that only one process executes in its critical section at a time, ensuring data consistency. Three requirements must ALL be satisfied: Mutual Exclusion, Progress, and Bounded Waiting.
Key Terms
TermDefinition
Race ConditionOutcome depends on the order/timing of concurrent process execution
Critical Section (CS)Code segment that accesses shared resources
Mutual ExclusionOnly one process can be in CS at a time
ProgressIf no process is in CS, a process wanting to enter must eventually be allowed
Bounded WaitingLimit on how many times others can enter CS before a waiting process gets in
MutexBinary mutual exclusion lock with ownership semantics
SemaphoreInteger variable; accessed only via atomic wait()/signal() operations
MonitorHigh-level construct with automatic mutual exclusion + condition variables
Busy Waiting / SpinlockProcess repeatedly checks condition without sleeping (wastes CPU)
DeadlockCircular wait — all processes blocked forever
LivelockProcesses change state but make no progress
Diagrams
CRITICAL SECTION REQUIREMENTS
──────────────────────────────────────────────────────────
 1. MUTUAL EXCLUSION
    If Pi is in CS, NO other process can be in CS

 2. PROGRESS
    If no process is in CS and some want to enter,
    only processes NOT in remainder section decide,
    and the decision cannot be postponed indefinitely

 3. BOUNDED WAITING
    There is a FINITE BOUND on how many times others
    can enter CS after a process has requested entry

CRITICAL SECTION TEMPLATE:
────────────────────────────
  do {
    [ ENTRY SECTION ]      ← request permission
    [ CRITICAL SECTION ]   ← access shared resource
    [ EXIT SECTION ]       ← signal CS is free
    [ REMAINDER SECTION ]  ← non-critical code
  } while (true);
RACE CONDITION — Example
──────────────────────────────────────────────────────────
Shared: count = 5

Process P1               Process P2
────────────────         ────────────────
reg1 = count (=5)        reg2 = count (=5)
reg1 = reg1 + 1          reg2 = reg2 + 1
reg1 = 6                 reg2 = 6
count = reg1 (=6)        count = reg2 (=6)

Expected: count = 7
Actual:   count = 6  ← RACE CONDITION! Lost update!
PETERSON'S SOLUTION (2 processes, software)
──────────────────────────────────────────────────────────
Shared: int turn; bool flag[2];

Process Pi:
  flag[i] = true;          // I want to enter
  turn = j;                // but give priority to Pj
  while (flag[j] && turn == j)
      ;                    // busy wait
  — CRITICAL SECTION —
  flag[i] = false;         // I'm done

✅ Satisfies all 3 conditions
❌ Busy waiting (wastes CPU)
❌ May not work on modern CPUs without memory barriers
SEMAPHORE — Operations
──────────────────────────────────────────────────────────
Semaphore S = integer variable

wait(S):              signal(S):
  while (S <= 0)        S++;
    ;  // busy wait
  S--;

Binary Semaphore (Mutex): S ∈ {0, 1}
Counting Semaphore:       S ∈ [0, N]

MUTEX USAGE:
  semaphore mutex = 1;
  wait(mutex);    // acquire lock
  — CRITICAL SECTION —
  signal(mutex);  // release lock

PRODUCER-CONSUMER (Bounded Buffer):
  semaphore mutex=1, empty=N, full=0;

  Producer:                Consumer:
    wait(empty);             wait(full);
    wait(mutex);             wait(mutex);
    // add item               // remove item
    signal(mutex);           signal(mutex);
    signal(full);            signal(empty);

  ⚠ ORDER MATTERS: always wait(empty/full) BEFORE wait(mutex)
     Reversing causes DEADLOCK!
DINING PHILOSOPHERS — The Classic Deadlock Problem
──────────────────────────────────────────────────────────
       P0
    f4    f0
  P4        P1
    f3    f1
       P3
    f2
       P2  (5 philosophers, 5 forks)

Naive: each picks left fork → ALL wait for right fork → DEADLOCK!

Solutions:

  2. Asymmetric: some pick left first, some pick right first

Real-World Analogy
Critical Section = Single-occupancy bathroom
Only one person at a time (mutual exclusion). If free and you're waiting, you should get in (progress). You can't wait forever if others keep jumping ahead (bounded waiting).

Semaphore = Parking lot with N spaces
wait() = take a slot (decrement). signal() = leave a slot (increment). If full (count=0), wait outside.

Monitor = Bank manager's office
Only one customer inside (automatic mutual exclusion). You wait on a chair (condition variable wait). Manager calls you in (condition variable signal).

Viva Q&A
Q: What is a race condition?
When the outcome of a program depends on the relative timing of concurrent processes. The processes "race" to access shared data, and the result varies unpredictably based on scheduling order.
Q: Difference between binary semaphore and mutex?
Binary semaphore: ANY process can call signal() on it, even one that didn't call wait(). Mutex: only the process/thread that LOCKED it can unlock it (ownership semantics). Mutex is stricter.
Q: Difference between semaphore and monitor?
Semaphore is a low-level primitive requiring manual placement of wait/signal. Monitor is high-level: mutual exclusion is AUTOMATICALLY enforced — programmer only handles condition synchronization (x.wait(), x.signal()).
Q: What is busy waiting? When is it acceptable?
Busy waiting = process loops checking condition without sleeping, wasting CPU cycles. Acceptable ONLY when wait time is expected to be very short (e.g., spinlocks in OS kernel for multicore systems where context switch overhead > wait time).
Q: What is the difference between deadlock and starvation?
Deadlock: processes permanently blocked — NONE can proceed (circular dependency). Starvation: process CAN proceed theoretically but NEVER gets the chance due to other processes always being preferred. A deadlocked process is also starved, but not vice versa.
Q: What is the difference between deadlock and livelock?
Deadlock: processes are completely blocked, NO state change. Livelock: processes ARE changing state but make no progress — like two people in a hallway each stepping the same direction to avoid the other, forever.
Trap Questions ⚠
Q: Does Peterson's solution work on modern hardware?
NOT without memory barriers! Modern CPUs reorder instructions for performance. The reads/writes to flag and turn may be reordered, violating the algorithm. Memory fence instructions are required. Peterson's is educational only.
Q: Can semaphores solve deadlock?
No — incorrect use of semaphores can actually CREATE deadlock (two processes each waiting for the other's semaphore). Deadlock prevention/avoidance requires separate strategies entirely.
Q: In Producer-Consumer, what happens if you acquire mutex BEFORE empty/full?
DEADLOCK. If producer holds mutex and then waits on empty (which is 0), and consumer needs mutex to signal empty — both are blocked permanently. Always acquire resource semaphores FIRST, then mutex.
Common Mistakes
Acquiring mutex BEFORE empty/full in Producer-Consumer
Always: wait(empty/full) first, THEN wait(mutex)
"Monitor = Semaphore"
Monitor provides AUTOMATIC mutual exclusion; semaphore is manual
Forgetting signal() after CS
Causes PERMANENT blocking of all waiting processes
"Peterson's is used in real OS"
Real OS uses hardware-atomic Compare-And-Swap (CAS) operations
1-Minute Revision
  • Race Condition: shared data + concurrent access = unpredictable result
  • 3 CS Requirements: Mutual Exclusion + Progress + Bounded Waiting
  • Peterson's: educational, busy wait, not safe on modern CPUs
  • Semaphore: integer + wait(P) + signal(V); binary or counting
  • Mutex vs Semaphore: mutex has OWNERSHIP; binary semaphore doesn't
  • Monitor: high-level, auto mutual exclusion, condition variables (x.wait/x.signal)
  • Producer-Consumer: wait(empty/full) BEFORE wait(mutex) — critical order
  • Classic problems: Producer-Consumer, Readers-Writers, Dining Philosophers
TOPIC 04

Deadlocks

Core Concept
A deadlock is a situation where a set of processes are all blocked, each waiting for a resource held by another process in the set — resulting in a circular wait with no process able to proceed. ALL four Coffman conditions must hold simultaneously for deadlock to occur. The OS must either prevent, avoid, detect, or recover from deadlocks.
Key Terms
TermDefinition
DeadlockSet of processes permanently blocked in circular resource dependency
Preemptable resourceCan be taken from process without failure (e.g., CPU, memory)
Non-preemptable resourceCannot be taken without causing failure (e.g., printer, mutex)
RAGResource Allocation Graph — directed graph showing assignments and requests
Safe StateThere exists at least one safe sequence of process execution
Safe SequenceOrder where each process can complete using available + held resources
Banker's AlgorithmDeadlock avoidance: checks safe state before each resource grant
Wait-For GraphRAG simplified to show only process→process dependencies
Four Coffman Conditions
ALL FOUR must hold simultaneously for deadlock:

┌─────────────────────────────────────────────────────────┐
│  1. MUTUAL EXCLUSION                                    │
│     At least one resource is non-shareable              │
│                                                         │
│  2. HOLD AND WAIT                                       │
│     Process holds ≥1 resource while waiting for more    │
│                                                         │
│  3. NO PREEMPTION                                       │
│     Resources cannot be forcibly taken from a process   │
│                                                         │

Real-World Analogy
Deadlock = Two-lane bridge blockage
Two cars meet head-on. Car A won't reverse until Car B does (hold & wait + circular wait). Bridge fits one car (mutual exclusion). Neither can be moved (no preemption). Result: neither moves — forever.

Banker's Algorithm = A careful banker
Has limited cash. Before lending, checks: "If I lend this, can I still guarantee everyone gets repaid?" Only lends when safe — refuses even if funds are available.
Viva Q&A
Q: What are the four conditions for deadlock?
Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait — ALL FOUR must hold simultaneously. Eliminating even one prevents deadlock.
Q: Is an unsafe state always a deadlock state?
NO! An unsafe state means deadlock MAY occur depending on future requests. A safe state guarantees deadlock WILL NOT occur. Deadlock is a subset of unsafe states — not all unsafe states lead to deadlock.
Q: What does a cycle in the RAG mean?
Single-instance resources: cycle = deadlock (necessary AND sufficient). Multiple-instance resources: cycle is NECESSARY but NOT sufficient — there may or may not be a deadlock. Need the detection algorithm to confirm.
Q: Difference between deadlock prevention and avoidance?
Prevention: eliminates one of the 4 conditions STATICALLY (restricts how resources are requested — before any execution). Avoidance: DYNAMICALLY examines each request and only grants if safe state is maintained (Banker's Algorithm).
Q: Why is the Ostrich Algorithm used in practice?
Deadlocks are rare, and prevention/avoidance mechanisms are expensive (both in performance and programming complexity). For general-purpose OS like UNIX/Linux, it's more practical to just reboot when a rare deadlock occurs than to pay constant overhead for every resource request.
Q: Can a single process deadlock with itself?
Normally no. But a process CAN deadlock by trying to acquire the same non-reentrant mutex twice (self-deadlock). The thread blocks waiting for a lock it already holds — permanent block.
Trap Questions ⚠
Q: If we break Mutual Exclusion, can we prevent all deadlocks?
No — you can't break mutual exclusion for non-shareable resources (like a printer). It only works for truly shareable resources (read-only files). This approach is very limited and cannot prevent all deadlocks.
Q: Banker's guarantees deadlock won't occur — true or false?
Partially true. Banker's guarantees the system won't enter an UNSAFE STATE based on declared maximum demands. If a process's actual behavior deviates from its declared maximum, the guarantee breaks. It also requires processes to declare max resources upfront, which isn't always practical.
Q: Can deadlock occur with preemptable resources?
No — if resources can be preempted, the OS can always break a circular wait by forcibly taking a resource. Deadlock only occurs with NON-PREEMPTABLE resources.
Common Mistakes
"Cycle in RAG always means deadlock"
Cycle = deadlock ONLY for single-instance resources
"Unsafe state = deadlock"
Unsafe = deadlock MAY happen; safe = deadlock WON'T happen
"Prevention and avoidance are the same"
Prevention = static (breaks conditions); Avoidance = dynamic (per-request check)
Forgetting Need = Max − Allocation
Need must be calculated first; it's the key input to Banker's
"All 4 conditions independently cause deadlock"
ALL FOUR must hold SIMULTANEOUSLY
1-Minute Revision
  • Deadlock: circular wait, all processes blocked permanently
  • 4 Conditions (ALL must hold): Mutual Exclusion + Hold & Wait + No Preemption + Circular Wait
  • RAG: request edge (P→R), assignment edge (R→P)
  • Cycle + single instance → deadlock; multi-instance → run detection algo
  • Prevention: break one condition (static) | Avoidance: Banker's per request (dynamic)
  • Safe state ≠ deadlock; Unsafe state ≠ guaranteed deadlock
  • Banker's: Need[i] = Max[i] − Allocation[i]; find safe sequence
  • Recovery: abort processes OR preempt resources (risk: starvation of victim)
TOPIC 05

Memory Management (Paging + Virtual Memory)

Core Concept
Memory management allows multiple processes to share physical memory safely. Paging divides memory into fixed-size blocks, eliminating external fragmentation. Virtual memory extends apparent memory beyond physical RAM using disk — allowing programs larger than RAM to execute via demand paging. The MMU translates logical addresses to physical addresses using the page table.

Page Replacement Algorithms
ALGORITHMS SUMMARY:

FIFO: Replace page in memory the LONGEST
  + Simple | - Belady's Anomaly | - Poor performance

OPT: Replace page not used for LONGEST TIME IN FUTURE
  + Minimum faults (benchmark) | - Requires future knowledge (impractical)

LRU: Replace page not used for LONGEST TIME IN PAST
  + Near-optimal | - Expensive to implement exactly | - No Belady's anomaly

CLOCK (2nd Chance): Approximation of LRU
  + O(1) | + Used in real OS | - Approximation only
  Each page has reference bit R.
  On access: R=1. On replacement: if R=1 → clear to 0 (second chance);
  if R=0 → replace. Clock hand sweeps pages in order.

Belady's Anomaly: ONLY affects FIFO, NOT LRU or OPT (stack algorithms)

Paging = Library book management
Books (pages) fit fixed-size shelves (frames). Your catalog (page table) says where each book is. The librarian (MMU) translates "I want book 3" → "it's on shelf 7, slot 4".

Virtual Memory = Desk + Filing cabinet
Your desk (RAM) is small. Filing cabinet (swap/disk) stores other documents. When you need a document not on your desk (page fault), you fetch it. If desk is full, file away the least-recently-used document (page replacement).

TLB = Sticky notes on your monitor
Instead of checking the full catalog every time, frequently-used book locations are on sticky notes (TLB). Cache miss = note not there, must check full catalog.

Viva Q&A
Q: What is the difference between logical and physical address?
Logical (virtual) address is generated by the CPU — what the process sees. Physical address is the actual RAM location. The MMU hardware translates logical → physical at runtime using the page table.
Q: What happens during a page fault?
CPU generates a trap → OS takes control → validates the reference (invalid = SIGSEGV, terminate) → finds free frame (evicts a page if none free, using replacement algorithm) → loads page from disk → updates page table (valid=1) → restarts the faulting instruction. Transparent to the process.
Q: Why does TLB improve performance so significantly?
Without TLB, every memory access requires 2 RAM accesses: one to the page table, one for the actual data. TLB caches recent page-to-frame mappings in the CPU. With 95-99% hit rate, effective access time approaches single-access time, nearly doubling memory throughput.
Q: What is Belady's Anomaly?
In FIFO page replacement, increasing the number of frames can INCREASE page faults — counterintuitive! This paradox doesn't affect LRU or OPT because they have the stack property (a set with n frames is a subset of the set with n+1 frames).
Q: What is thrashing and how is it prevented?
Thrashing occurs when a process has fewer frames than its working set — it generates page faults continuously and spends more time swapping than executing. Prevention: Working Set Model (ensure each process has enough frames for its working set), or Page Fault Frequency control (adjust frames based on fault rate).
Q: What is Copy-on-Write (COW)?
After fork(), parent and child share the same physical pages (mapped read-only). When EITHER writes to a page, the OS creates a private copy of just that page for the writing process. Avoids unnecessary copying of all pages — critical optimization for fork() performance.
Q: Difference between page fault and segmentation fault?
Page fault: page is valid but NOT in RAM — OS handles transparently by loading from disk, then restarts the instruction. Segmentation fault: process accessed an INVALID or unauthorized address — OS terminates the process with SIGSEGV. Page fault is normal; segfault is an error.
Trap Questions ⚠
Q: Why can't we implement true LRU in hardware?
True LRU needs either: a hardware counter updated on EVERY memory access (time-stamps every access — too expensive), or a stack of pages reordered on every access (too slow). Hardware only provides a single reference bit, which the OS uses to approximate LRU via the Clock algorithm.
Q: Can page size affect performance?
Yes. Large pages: fewer page table entries, fewer TLB entries needed → better TLB hit rate, but more internal fragmentation and larger page fault penalty (more data to load). Small pages: less fragmentation, more TLB pressure. Most OS use 4KB; some use huge pages (2MB/1GB) for specific workloads like databases.
Q: Is TLB in RAM?
No! TLB is in CPU HARDWARE (on-chip associative cache), not RAM. That's why TLB hits are so fast (1-2 cycles). Page tables are in RAM — that's why TLB misses require additional memory accesses.
